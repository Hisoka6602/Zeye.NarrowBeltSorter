# 包裹密集上车小车号偏差根因分析

## 问题描述

在分拣系统运行过程中，当包裹创建越密集（上包频率越高）时，分拣系统为包裹分配的上车小车号就越容易出现偏差，进而导致落格位置错误。

---

## 系统链路简述

理解根因之前，需先梳理从"传感器触发"到"小车号写入包裹"的完整链路：

```
FirstCarSensor / NonFirstCarSensor 触发
        │
        ▼
CarrierLoopGroupingHostedService.OnSensorStateChanged
  └── _safeExecutor.Execute(() => HandleSensorStateChanged(args))  ← 同步
           │
           ├── lock(_counterLock) 计算 currentCarrierId
           │
           └── [lock 外] fire-and-forget:
                ├── _safeExecutor.ExecuteAsync(() => BuildRingAsync(...))        ← 异步①
                └── _safeExecutor.ExecuteAsync(() => UpdateCurrentInductionCarrierAsync(id)) ← 异步②
                         │
                         ▼
              InfraredSensorCarrierManager.UpdateCurrentInductionCarrierAsync
                  └── _safeExecutor.PublishEventAsync(CurrentInductionCarrierChanged, ...)
                           │  ← SafeExecutor.PublishEventAsync 对每个订阅者调用
                           │    ThreadPool.UnsafeQueueUserWorkItem 并行分发
                           ▼
              SortingTaskDropOrchestrationService.HandleCurrentInductionCarrierChangedAsync
                  └── SortingTaskCarrierLoadingService.TryLoadParcelAtLoadingZoneAsync(newCarrierId)
                           │
                           ▼
                  CircularValueHelper.GetCounterClockwiseValue(newCarrierId, offset, total)
                           │
                           ▼
                  loadingCarrier.LoadParcelAsync(parcel, ...)  ← 包裹绑定上车
```

---

## 根因一：fire-and-forget 并发导致 UpdateCurrentInductionCarrier 乱序到达

### 代码定位

`CarrierLoopGroupingHostedService.HandleSensorStateChanged`（原始实现）：

```csharp
lock (_counterLock) {
    // ... 根据触发顺序更新 _currentRingIndex 和 currentCarrierId
}

// lock 外：两个 fire-and-forget 异步任务
if (currentCarrierId > 0) {
    _ = _safeExecutor.ExecuteAsync(
        async () => {
            await _carrierManager.UpdateCurrentInductionCarrierAsync(currentCarrierId);
        },
        "...");
}
```

### 乱序时序示例

假设传感器以极短间隔连续触发两次：

```
时间轴 ────────────────────────────────────────────────────────►

线程A（触发#5, carrierId=5）:
  [lock] _currentRingIndex=4, currentCarrierId=5 [unlock]
  ─── 发起 Task_A: UpdateCurrentInductionCarrier(5) ──────────────►
                                                               完成: CurrentId=5

线程B（触发#6, carrierId=6，几乎同时）:
  [lock] _currentRingIndex=5, currentCarrierId=6 [unlock]
  ─── 发起 Task_B: UpdateCurrentInductionCarrier(6) ──────►
                                                   完成: CurrentId=6
                                                               之后:
                                                   CurrentId=6 （正确）
                                                               然后 Task_A 完成:
                                                               CurrentId=5 ← 错误！
```

**结论**：`lock` 内 `_currentRingIndex` 的递增顺序是正确的，但两个异步任务完成的先后顺序不受控制。若 Task_B（较晚的触发）先完成，Task_A（较早的触发）后完成，则最终写入的 `CurrentInductionCarrierId` 是旧值，导致之后所有依赖该值的上车/落格计算全部使用错误的小车号。

### 为何密度越高偏差越大

| 上包频率 | 单位时间传感器触发次数 | 并发 fire-and-forget 任务数 | 乱序概率 |
|---------|---------------------|--------------------------|---------|
| 低密度   | 少（间隔 ≥ 500ms）   | 极少同时并发执行               | 极低     |
| 高密度   | 多（间隔 ≤ 100ms）   | 大量同时并发执行               | 显著升高  |

间隔越短，两个任务的"发起时间"越接近，ThreadPool 调度的不确定性就越容易将它们的完成顺序打乱。高密度时多个任务同时并发执行，乱序叠加效应更为剧烈，**最终记录的小车号可能落后当前实际位置 2~N 格，导致装车到错误小车、格口触发时序偏移**。

## 根因二：建环闭合时 BuildRing 与 UpdateCurrentInductionCarrier 的竞争

### 触发场景

在系统完成建环的瞬间（首车传感器第二次触发），原始代码同时 fire-and-forget 两个任务：

```csharp
// 同时启动，执行顺序不确定
_ = _safeExecutor.ExecuteAsync(() => BuildRingAsync(ringClosedCarrierIds, ...));
_ = _safeExecutor.ExecuteAsync(() => UpdateCurrentInductionCarrierAsync(currentCarrierId));
```

### 竞争时序示例

```
情形 A（正常顺序）:
  Task_Build  → IsRingBuilt=true, Carriers=[1..N]
  Task_Update → CurrentInductionCarrierId=1 → 触发 CurrentInductionCarrierChanged
              → TryLoadParcelAtLoadingZoneAsync 能获取 Carriers.Count=N（正确）

情形 B（乱序）:
  Task_Update → CurrentInductionCarrierId=1 → 触发 CurrentInductionCarrierChanged
              → TryLoadParcelAtLoadingZoneAsync 读到 IsRingBuilt=false 或旧 Carriers.Count
              → 直接 return，包裹未被装载
  Task_Build  → IsRingBuilt=true, Carriers=[1..N]
              → （此次建环窗口内的上车机会已丢失）
```

**结论**：在建环瞬间存在一个"状态不一致窗口"，`CurrentInductionCarrierId` 已更新但 `Carriers` 环尚未重建，依赖该 ID 触发的上车/落格判断使用的是不完整的小车环，结果是包裹错装或漏装。密度越高，建环窗口内同时到达的成熟包裹越多，受影响的包裹数量越大。

## 根因三：上车触发的"相对位置"语义依赖小车号绝对正确性

### 上车偏移计算

```csharp
var loadingCarrierId = CircularValueHelper.GetCounterClockwiseValue(
    currentCarrierValue,         // 当前感应位小车编号
    _carrierManager.LoadingZoneCarrierOffset,  // 上车位偏移量（例如 3）
    totalCarrierCount);
```

感应位小车号一旦偏差 1 格，上车位小车号同样偏差 1 格，**包裹会被装到相邻小车**。若偏差 2 格，上车位错位 2 格。密度越高乱序越频繁，偏差范围越大。

### 落格偏移计算的连锁影响

落格使用同一个 `currentInductionCarrierId`：

```csharp
var carrierIdAtChute = ResolveCarrierIdAtChute(
    args.NewCarrierId.Value,   // 感应位小车号（若已乱序则已是错值）
    pair.Value,                // 格口偏移量
    orderedCarrierIds);
```

若感应位小车号错误，**落格目标小车也随之偏移**，导致包裹落入错误格口或完全漏格。

## 密度与偏差的放大关系总结

```
低密度（间隔充分）:
  触发A ──[完成]── 触发B ──[完成]── ...
  每次 Update 在下一次触发前完成，乱序概率趋近于零。

高密度（间隔极短）:
  触发A──触发B──触发C──...（多个触发几乎同时）
       ↓     ↓     ↓
  Task_A Task_B Task_C  ← 同时在 ThreadPool 队列中并发执行
  完成顺序: Task_C Task_A Task_B  ← 随机
  最终 CurrentId = Task_B 的值（不是最新的 Task_C）
  ↓
  上车偏差 = |C - B| 格
  落格偏差同步放大
```

偏差量 = 乱序幅度 × 格口偏移误差放大率，**密度越高乱序幅度越大，最终落格偏差越严重**。

## 可观测的日志征兆

在日志中可观察到以下异常模式（不修改代码的情况下辅助确认）：

| 日志关键字 | 含义 |
|----------|------|
| `CarrierLoopGrouping 触发类型=非首车触发-环运行 CurrentCarrierId=X` | X 的递增应严格连续；若出现跳变或倒退，说明乱序已发生 |
| `当前感应位小车Id=X` 后紧跟 `装车成功 CarrierId=Y`，X≠Y+offset | 上车时感应位与上车位偏移不符，说明感应位小车号已偏移 |
| `错过格口 ParcelId=P CarrierId=C TargetChuteId=T` | 包裹未能在正确格口落格，与小车号偏差直接相关 |
| `装车事件并发冲突，已存在映射回退包裹 CarrierId=C` | 同一小车被两次装载，通常源于连续触发的 CarrierId 相同（乱序导致重复写入同一小车号） |
