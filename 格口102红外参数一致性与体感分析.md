## 3. 与当前实现口径的逐项核对

> 说明：以下“是否一致”指“是否满足当前编码器实现可编码并下发”。

| 参数 | 实现口径 | 结果 |
|---|---|---|
| DinChannel=2 | `2 -> D2(0xD2)`，可映射 | ✅ 一致 |
| DefaultDirection=Right | 方向位由 `BuildByte2` 写入 | ✅ 一致 |
| ControlMode=Position | `Position` 走位置模式编码 | ✅ 一致 |
| DefaultDistanceMm=1000 | 位置模式会参与 `runRaw` 计算，1000 可编码 | ✅ 一致 |
| AccelerationMmps2=5000 | 实现允许范围 `1500~7850`，5000 可编码 | ✅ 一致 |
| TriggerDelayMs=0 | 允许 `>=0`，0 可编码 | ✅ 一致 |
| RollerDiameterMm=67 | `>0` 可用于速度/位移换算 | ✅ 一致 |
| DialCode=16 | Byte2 仅取低 6 位，16 有效 | ✅ 一致 |
| HoldDurationMs=60 | 当前编码器未使用该字段 | ⚠️ 配置可写入但对当前下发帧无影响 |
| **DefaultSpeedMmps=5000** | 速度需换算为 7bit 原始值（0~127）；5000 会超上限 | ❌ **不一致（会导致整帧编码失败）** |

---

## 4. 关键原因：为什么“更新后速度、时间变化不大”

### 4.1 速度 5000 在当前实现下会编码失败（核心原因）

当前编码器速度换算链路：

1. `mm/s -> rpm -> raw`
2. `raw` 必须为 7bit（0~127）

按实现常量（`RpmPerSpeedRaw=6`，`ProtocolPi=3.14`，`RollerDiameterMm=67`）计算：

- 周长 `C = 67 * 3.14 = 210.38 mm`
- `rpm = 5000 * 60 / 210.38 ≈ 1425.99`
- `speedRaw = rpm / 6 ≈ 237.67`

而 `237.67 > 127`，`TryConvertTo7Bit` 返回失败，导致 `EncodeCore` 返回失败，最终整帧不下发。  
**因此“改到 5000 后看起来几乎没变化”是符合当前代码行为的。**

---

### 4.2 `HoldDurationMs` 当前未进入编码链路

`InfraredChuteOptions` 虽有 `HoldDurationMs` 字段，但当前 `LeadshaineInfraredDriverFrameCodec` 编码路径未使用该字段。  
所以 `50 -> 60` 在当前实现中不会体现在下发帧上，体感自然不明显。

---

### 4.3 位置模式下未修改 `DefaultDistanceMm`，时间体感通常不敏感

当前是 `ControlMode=Position` 且 `DefaultDistanceMm=1000` 不变。  
在位置模式中，动作“跑多远”主要受距离参数影响；若距离不变、触发延时仍为 0，则动作总时长变化通常不会像“直接改时长模式参数”那样明显。

---

### 4.4 热更新失败后，当前逻辑仍会刷新签名，导致不再自动重试

`ChuteSelfHandlingHostedService` 在遍历下发后（即使某些格口下发失败）仍会更新 `_lastInfraredSignature`。  
这会导致“同一份失败配置”在后续周期不再重试，除非配置再次变更。  
现场感知就是：改完一次没明显变化，后面也不会自动继续尝试。

---

## 5. 当前仓库里可直接观察的现象

1. 修改为 `DefaultSpeedMmps=5000` 后，格口 102 会出现“热更新失败/写入失败”日志。  
2. 若仅改 `HoldDurationMs`，通常不会观察到动作差异（当前编码未使用）。  
3. 若改完后无再次配置变更，失败配置可能不会自动重复下发（签名已更新）。

---

## 6. 建议（不改代码前提下）

1. 先把 102 的 `DefaultSpeedMmps` 调整到当前实现可编码范围内（按 67mm 直径，建议明显低于 5000）。  
2. 需要显著改变动作时长时，优先联动调整 `ControlMode/DefaultDistanceMm/TriggerDelayMs`，而不是只改 `HoldDurationMs`。  
3. 每次修改后观察“成功/失败”日志，再判断机械体感，避免“配置已改但帧未成功下发”的误判。
