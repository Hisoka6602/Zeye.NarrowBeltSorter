# 逐文件全量审查实施方案（可拆分多个 PR）

## 1. 目标与范围

- 目标：对仓库进行逐文件全覆盖审查，识别并治理以下问题：
  - 影分身代码（重复实现/复制粘贴逻辑）
  - 过度设计与冗余代码
  - 性能问题（含热路径低效实现）
  - 逻辑正确性问题
  - 并发竞态与线程安全问题
  - `.github/copilot-instructions.md` 规则违规项
- 范围：仓库内全部源代码、配置、文档与 CI 规则脚本（按分层与模块逐批审查）。
- 约束：本方案用于“只审查不改代码”阶段；后续按批次 PR 落地修复。

## 2. 审查总原则

- 必须逐文件打卡，禁止“按目录抽样”导致遗漏。
- 每个文件至少经过一次“规则合规 + 质量 + 风险”三维审查。
- 每批次 PR 只处理单一主题（如并发、日志、配置、命名），降低回归风险。
- 发现问题必须落在可追溯清单（文件路径 + 行号 + 问题类型 + 严重级别 + 修复建议）。
- 同类问题优先抽取公共能力，禁止新增影分身修复代码。

## 3. 分批审查顺序（建议 8 批）

1. **批次 A：规则与基线**
   - `.github/copilot-instructions.md`
   - `.github/workflows/*`
   - `.github/scripts/*`
   - 根目录治理文档（`README.md`、结构清单等）
2. **批次 B：Core 抽象层**
   - `Zeye.NarrowBeltSorter.Core/Manager/**`
   - `Zeye.NarrowBeltSorter.Core/Enums/**`
   - `Zeye.NarrowBeltSorter.Core/Events/**`
   - `Zeye.NarrowBeltSorter.Core/Options/**`
   - `Zeye.NarrowBeltSorter.Core/Utilities/**`
3. **批次 C：Execution 业务编排层**
   - `Zeye.NarrowBeltSorter.Execution/**`
4. **批次 D：Drivers 设备驱动层**
   - `Zeye.NarrowBeltSorter.Drivers/**`
5. **批次 E：Host/Ingress/Infrastructure 承载层**
   - `Zeye.NarrowBeltSorter.Host/**`
   - `Zeye.NarrowBeltSorter.Ingress/**`
   - `Zeye.NarrowBeltSorter.Infrastructure/**`
6. **批次 F：测试层**
   - `Zeye.NarrowBeltSorter.Core.Tests/**`
7. **批次 G：配置层**
   - `appsettings*.json`
8. **批次 H：文档与清单一致性**
   - 根目录 `*.md`
   - `Manager接口结构清单.md`
   - `设备代码结构清单.md`
   - `README.md` 文件树与职责说明

## 4. 逐文件检查项（每个文件都要过）

### 4.1 影分身代码与冗余

- 是否存在同语义重复方法/重复分支/重复常量拼装。
- 是否存在可抽取的公共工具却分散实现。
- 是否存在无效字段、无效参数、无效状态变量。

### 4.2 过度设计

- 是否存在“只用一次”的抽象层、泛化接口或过深继承链。
- 是否引入与当前场景不匹配的复杂机制（策略叠加、状态机过度拆分等）。
- 是否出现跨层侵入导致层级职责边界模糊或耦合。

### 4.3 性能

- 热路径是否有高频分配、重复查找、重复解析、阻塞 I/O。
- 日志是否在高频路径做昂贵字符串拼接且无级别守卫。
- 配置/数据库是否在热路径被频繁读写（禁止项）。
- 是否可利用现有高性能特性但未使用。

### 4.4 逻辑正确性

- 空值、边界值、状态切换顺序、错误分支是否完整。
- 是否存在仅记录日志但未中断流程或执行补偿逻辑的隐性问题。
- 设备协议实现是否与文档一致，是否存在低效往返读写。

### 4.5 并发与竞态

- 共享状态是否具备并发保护与可见性保证。
- 事件发布是否非阻塞、订阅者是否并行且不拖垮发布者。
- 是否存在订阅泄漏、重复订阅、竞态覆盖、乱序处理风险。

### 4.6 规则合规（`.github/copilot-instructions.md`）

- 时间语义：统一使用本地时间语义，禁止引入本地时间之外的时间语义与相关 API。
- 枚举位置与注释：必须在 `Core/Enums` 子目录，且包含 `Description` 与注释。
- 事件载荷：必须在 `Events` 子目录，且使用 `readonly record struct`。
- 异常路径：必须记录并落盘日志，且日志框架仅允许 NLog。
- 安全执行：危险代码统一通过 `SafeExecutor`。
- Options/interface/Utilities 的目录约束必须满足。
- 配置注释需中文且写明范围；枚举配置需列出所有枚举项。
- Host 层目录命名使用 `Services`，禁止 `Servers`。
- 禁止触发重放机制，遵循“先有包裹才有触发”。

## 5. 执行节奏（每个批次 PR）

1. 建立该批次文件清单（必须可勾选）。
2. 逐文件审查并记录问题。
3. 完成“遗漏复核”（清单必须全部勾选）。
4. 输出该批次审查报告（不改代码或附修复 PR 链接）。
5. 进入下一批次。

## 6. 审查记录模板（复制即用）

| 文件路径 | 已审查 | 问题类型 | 严重级别 | 位置（行号） | 问题描述 | 修复建议 | 计划 PR |
|---|---|---|---|---|---|---|---|
| `示例/文件.cs` | [ ] | 影分身/性能/并发/规则 | 高/中/低 | 120-180 |  |  | `PR-批次X-序号` |

> 规则：`已审查` 未勾选即视为未完成，批次不得关闭。
> 规则：行号应尽量精确定位问题代码段，建议单条记录跨度不超过 20 行。

## 7. 验收门禁（审查阶段）

- [ ] 该批次文件清单 100% 勾选完成。
- [ ] 每个问题都有文件路径、行号、类型、严重级别。
- [ ] 已完成“重复问题归并”，避免同类问题重复开单。
- [ ] 已标注可并行修复与必须串行修复项。
- [ ] 已给出下一批次边界与入口文件清单。

## 8. 多 PR 拆分策略

- 一个 PR 只处理一个批次，避免审查与修复耦合过深。
- 大批次可再按子域拆分（如 Execution 按 `Services/`、`Parcel/`、`State/`）。
- 若单批问题过多，先提“审查结果 PR”，再按问题类型拆“修复 PR”。

## 9. 待确认项（执行前必须确认）

- 是否要求“审查 PR”与“修复 PR”严格分离（建议分离）。
- 严重级别判定是否采用三档（高/中/低）或四档（致命/高/中/低）。
- 每批次是否要求固定最大文件数（例如 80 文件封顶）。
