# 配置文件拆分分析

## 当前策略

当前仓库采用“按能力拆分 + 全环境统一基线”策略，不再使用 `appsettings.{Env}*.json` 环境覆盖文件。  
无论是否 `DEBUG`，均加载同一套正式参数文件运行。

---

## 拆分边界原则

| 原则 | 说明 |
|------|------|
| **能力优先** | 每个 JSON 文件对应一个独立的运行能力模块 |
| **设备参数单独管理** | 串口号、IP、点位映射等硬件参数收敛到 `appsettings.devices.{module}.json` |
| **全环境统一参数** | 删除 Development 覆盖文件后，所有环境共享一套参数，避免环境分歧 |

---

## 当前文件结构与职责

### 加载顺序（低优先级 → 高优先级，后加载覆盖先加载）

```text
appsettings.json                        # 全局基础配置（日志清理、分拣时序、日志级别）
appsettings.looptrack.json              # LoopTrack 能力配置
appsettings.chutes.json                 # Chutes 与 Carrier 能力配置
appsettings.leadshaine.json             # Leadshaine EMC/IoPanel/Sensor/SignalTower/IoLinkage 配置
appsettings.chutes.devices.json         # Chutes 设备参数（IP/端口/红外映射/格口映射）
环境变量（最高优先级）
命令行参数（最高优先级）
```

若 `ZEYE_USE_ENV_ONLY_CONFIG=true`，跳过所有 JSON 文件，仅使用环境变量与命令行参数。

---

## 实现说明

`Zeye.NarrowBeltSorter.Host/Vendors/DependencyInjection/HostApplicationBuilderConfigurationExtensions.cs`  
已移除 `appsettings.{env}.json`、`appsettings.{env}.looptrack.json`、`appsettings.{env}.chutes.json`、`appsettings.{env}.leadshaine.json`、`appsettings.{env}.devices.looptrack.json`、`appsettings.{env}.devices.chutes.json` 的加载逻辑。

---

## 后续建议

1. 若后续确需差异化参数，优先通过环境变量/命令行覆盖，避免回到多套 JSON 分叉。
2. 若新增能力模块，按同样边界新增 `appsettings.{module}.json` 与可选 `appsettings.devices.{module}.json`，保持结构一致。
