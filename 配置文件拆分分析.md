# 配置文件拆分分析

## 问题历史

`appsettings.Development.json` 曾包含约 685 行内容，将以下完全不同职责的配置混放在单一文件中：

| 配置键 | 行数 | 职责 |
|--------|------|------|
| `LogCleanup` | ~10 | 日志清理开关 |
| `LoopTrack` | ~140 | 环轨驱动（LeiMa）完整参数 |
| `Chutes` | ~60 | 智嵌格口驱动参数 |
| `Carrier` | ~10 | 小车建环与偏移映射 |
| `Leadshaine` | ~445 | EMC 连接、点位绑定、IoPanel、Sensor、SignalTower、IoLinkage |
| `Logging` | ~8 | .NET 日志级别 |

单文件过大曾导致：

- 修改任意一项配置都要在 685 行中定位，维护成本高。
- 不同角色（调试人员/环境配置/设备调试）需要的配置交织在一起，边界不清晰。
- 在 Git 历史中无法快速区分"改了设备参数"还是"改了业务开关"。

---

## 拆分边界原则

拆分以"能力模块"为边界，每个文件只负责一类独立功能域：

| 原则 | 说明 |
|------|------|
| **能力优先** | 每个 JSON 文件对应一个独立的运行能力模块，互不依赖 |
| **设备参数单独管理** | 串口号、IP、点位映射等硬件参数收敛到 `devices.{module}.json` |
| **环境覆盖按需添加** | 仅当开发环境需要覆盖不同于基础文件的值时，才新增 `{Env}.{module}.json`；否则直接使用基础文件 |
| **Development 与生产共用基础文件** | 基础文件的值同时满足 Development 与生产，无需重复维护两份相同内容 |

## 当前文件结构与职责

### 加载顺序（低优先级 → 高优先级，后加载的覆盖先加载的）

```text
appsettings.json                        # 全局基础默认（兼作 Development 环境基线）
appsettings.looptrack.json              # 全局环轨能力参数（可选）
appsettings.chutes.json                 # 全局格口能力参数（可选）
appsettings.leadshaine.json             # 全局 Leadshaine 能力参数（可选）
appsettings.devices.looptrack.json      # 全局环轨设备参数（串口/从站等硬件参数）
appsettings.devices.chutes.json         # 全局格口设备参数（IP/端口/格口映射/红外参数）
appsettings.{Env}.json                  # 环境基础覆盖（仅在需要覆盖值时添加）
appsettings.{Env}.looptrack.json        # 环境环轨覆盖（仅在需要覆盖 PID、速度目标等时添加）
appsettings.{Env}.chutes.json           # 环境格口覆盖（仅在需要覆盖 Chutes/Carrier 时添加）
appsettings.{Env}.leadshaine.json       # 环境 Leadshaine 覆盖（仅在需要覆盖点位绑定等时添加）
appsettings.{Env}.devices.looptrack.json # 环境环轨设备覆盖（仅在需要覆盖串口号等时添加）
appsettings.{Env}.devices.chutes.json   # 环境格口设备覆盖（仅在需要覆盖设备 IP 等时添加）
环境变量（最高优先级）
命令行参数（最高优先级）
```

### 各文件职责说明

#### `appsettings.json`（必须）
- 配置项模板与默认值，兼作 Development 与生产共用基线。
- 包含完整字段与中文注释（作为配置文档）。

#### `appsettings.{module}.json`（可选）
- 按能力模块拆分的基础参数：`looptrack`、`chutes`、`leadshaine`。
- 各模块参数相互隔离，修改一个模块不影响其他模块的 Git 历史。

#### `appsettings.devices.{module}.json`（可选）
- 设备硬件参数：串口 RTU 参数、ZhiQian 设备 TCP 地址、格口映射、红外参数等。
- 与能力模块参数分离，方便设备调试人员单独维护。

#### `appsettings.{Env}.*.json`（按需添加）
- 仅在当前环境需要覆盖基础文件中的值时才添加对应的环境覆盖文件。
- 若开发环境与基础文件值相同，无需创建对应的 Development 文件。

## ConfigureConfigurationSources 加载顺序实现

`HostApplicationBuilderConfigurationExtensions.ConfigureConfigurationSources` 按以下顺序加载（代码实现见 `Vendors/DependencyInjection/HostApplicationBuilderConfigurationExtensions.cs`）：

1. `appsettings.json`（必须存在）
2. `appsettings.looptrack.json`（optional，环轨能力模块）
3. `appsettings.chutes.json`（optional，格口能力模块）
4. `appsettings.leadshaine.json`（optional，Leadshaine 能力模块）
5. `appsettings.devices.looptrack.json`（optional，环轨设备参数）
6. `appsettings.devices.chutes.json`（optional，格口设备参数）
7. `appsettings.{Env}.json`（optional）
8. `appsettings.{Env}.looptrack.json`（optional）
9. `appsettings.{Env}.chutes.json`（optional）
10. `appsettings.{Env}.leadshaine.json`（optional）
11. `appsettings.{Env}.devices.looptrack.json`（optional）
12. `appsettings.{Env}.devices.chutes.json`（optional）
13. 环境变量
14. 命令行参数

若 `ZEYE_USE_ENV_ONLY_CONFIG=true`，跳过所有 JSON 文件（步骤 1~12），仅使用环境变量与命令行。

## 后续建议

1. 若增加新的能力模块（如西门子 S7），可按同样边界新增 `appsettings.siemens.json`，保持文件结构一致性；如需环境差异化再按需添加 `appsettings.{Env}.siemens.json`。
2. Leadshaine 点位绑定（PointBindings）因与设备强绑定，长期可考虑移入 `appsettings.devices.leadshaine.json`，与设备 IP/串口参数统一管理。
3. 各环境配置覆盖文件只需包含与基础文件不同的配置项；保持覆盖文件尽量小，便于 review 差异。
