# 配置文件拆分分析

## 问题分析

`appsettings.Development.json` 曾包含约 685 行内容，将以下完全不同职责的配置混放在单一文件中：

| 配置键 | 行数 | 职责 |
|--------|------|------|
| `LogCleanup` | ~10 | 日志清理开关 |
| `LoopTrack` | ~140 | 环轨驱动（LeiMa）完整参数 |
| `Chutes` | ~60 | 智嵌格口驱动参数 |
| `Carrier` | ~10 | 小车建环与偏移映射 |
| `Leadshaine` | ~445 | EMC 连接、点位绑定、IoPanel、Sensor、SignalTower、IoLinkage |
| `Logging` | ~8 | .NET 日志级别 |

单文件过大导致：

- 修改任意一项配置都要在 685 行中定位，维护成本高。
- 不同角色（调试人员/环境配置/设备调试）需要的配置交织在一起，边界不清晰。
- 在 Git 历史中无法快速区分"改了设备参数"还是"改了业务开关"。

---

## 拆分边界原则

拆分以"能力模块"为边界，每个文件只负责一类独立功能域：

| 原则 | 说明 |
|------|------|
| **能力优先** | 每个 JSON 文件对应一个独立的运行能力模块，互不依赖 |
| **环境覆盖独立** | 每个能力模块可按环境独立覆盖，不影响其他模块 |
| **设备参数单独管理** | 串口号、IP、点位映射等硬件参数收敛到 `*.devices.json` |
| **通用开关保留在主文件** | 只有全局开关（LogCleanup、Logging 级别）保留在 `appsettings.Development.json` |

---

## 新的文件结构与职责

### 加载顺序（低优先级 → 高优先级，后加载的覆盖先加载的）

```text
appsettings.json                        # 全局基础默认（模板值）
appsettings.devices.json                # 全局设备硬件参数
appsettings.looptrack.json              # 全局环轨能力参数（可选）
appsettings.chutes.json                 # 全局格口能力参数（可选）
appsettings.leadshaine.json             # 全局 Leadshaine 能力参数（可选）
appsettings.{Env}.json                  # 环境基础覆盖（LogCleanup、Logging 级别）
appsettings.{Env}.devices.json          # 环境设备覆盖（串口号、IP）
appsettings.{Env}.looptrack.json        # 环境环轨覆盖（PID、速度目标）
appsettings.{Env}.chutes.json           # 环境格口覆盖（Chutes、Carrier）
appsettings.{Env}.leadshaine.json       # 环境 Leadshaine 覆盖（点位绑定、IoPanel）
环境变量（最高优先级）
命令行参数（最高优先级）
```

### 各文件职责说明

#### `appsettings.json`（已有）
- 配置项模板与默认值。
- 所有开关默认为 `false`/`Enabled: false`，避免生产误启动。
- 包含完整字段与中文注释（作为配置文档）。

#### `appsettings.devices.json`（已有）
- 设备硬件参数：串口 RTU 参数、ZhiQian 设备 TCP 地址。
- 生产环境的设备参数基准值。

#### `appsettings.Development.json`（已精简）
- 仅包含开发环境通用开关：`LogCleanup`（RetentionDays=2）、`Logging`（日志级别）。
- **不包含**任何设备或能力模块参数。

#### `appsettings.Development.looptrack.json`（新建）
- 环轨（LoopTrack）开发环境覆盖：目标速度、PID 参数、HIL 联调开关、日志频率等。
- 修改环轨参数时只需改这一个文件。

#### `appsettings.Development.chutes.json`（新建）
- 格口（Chutes/ZhiQian）与小车（Carrier）开发环境覆盖。
- 包含：格口开关、轮询间隔、强排/落格模拟开关、小车偏移映射。

#### `appsettings.Development.leadshaine.json`（新建）
- Leadshaine EMC 开发环境覆盖：EMC 连接参数、点位绑定（PointBindings）、IoPanel 按钮、Sensor 绑定、SignalTower 点位、IoLinkage 规则。
- 点位绑定数量大，单独隔离后不影响其他模块的读取和 review。

#### `appsettings.Development.devices.json`（已有）
- 开发环境设备硬件参数：LeiMa 串口号（COM4）、ZhiQian 设备配置与格口映射。

## 实施结果

| 文件 | 拆分后行数 |
|------|-----------|
| `appsettings.Development.json` | 22 行 |
| `appsettings.Development.looptrack.json` | ~130 行 |
| `appsettings.Development.chutes.json` | ~80 行 |
| `appsettings.Development.leadshaine.json` | ~450 行 |
| `appsettings.Development.devices.json` | 已有 |

原 685 行的单文件已拆分为职责清晰的独立文件，每次改动只需定位到对应模块文件。

## ConfigureConfigurationSources 加载顺序实现

`HostApplicationBuilderConfigurationExtensions.ConfigureConfigurationSources` 按以下顺序加载（代码实现见 `Vendors/DependencyInjection/HostApplicationBuilderConfigurationExtensions.cs`）：

1. `appsettings.json`（必须存在）
2. `appsettings.devices.json`（optional）
3. `appsettings.looptrack.json`（optional，环轨能力模块）
4. `appsettings.chutes.json`（optional，格口能力模块）
5. `appsettings.leadshaine.json`（optional，Leadshaine 能力模块）
6. `appsettings.{Env}.json`（optional）
7. `appsettings.{Env}.devices.json`（optional）
8. `appsettings.{Env}.looptrack.json`（optional）
9. `appsettings.{Env}.chutes.json`（optional）
10. `appsettings.{Env}.leadshaine.json`（optional）
11. 环境变量
12. 命令行参数

若 `ZEYE_USE_ENV_ONLY_CONFIG=true`，跳过所有 JSON 文件（步骤 1~10），仅使用环境变量与命令行。

## 后续建议

1. 若 `appsettings.json` 也趋于臃肿，可按同样边界拆分为 `appsettings.looptrack.json`、`appsettings.chutes.json`、`appsettings.leadshaine.json`（基础默认值文件）。
2. Leadshaine 点位绑定（PointBindings）因与设备强绑定，长期可考虑移入 `appsettings.{Env}.devices.json`，与设备 IP/串口参数统一管理。
3. 若增加新的能力模块（如西门子 S7），可按同样边界新增 `appsettings.{Env}.siemens.json`，保持文件结构一致性。
