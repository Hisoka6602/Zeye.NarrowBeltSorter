## 3. IEmcController（SiemensS7EmcController）实现机制拆解

## 3.1 抽象契约层（Core）

`IEmcController` 暴露的核心能力：

- 生命周期：`InitializeAsync`、`ReconnectAsync`
- 点位管理：`SetMonitoredIoPointsAsync`
- IO 写入：`WriteIoAsync`
- 状态观测：`Status`、`FaultCode`、`MonitoredIoPoints`
- 事件：`StatusChanged`、`Faulted`、`Initialized`

该设计重点是把“设备连接 + IO 快照 + 故障隔离”统一到一个入口，业务层只消费快照与写入接口。

## 3.2 配置与寻址层（Vendor Options + Binding）

SiemensS7 不让业务配置直接写 `I0.1/Q0.3/DBx.DBXy.z`，而是走“逻辑点位 PointId -> S7 物理地址”映射：

- 连接参数：`S7EmcConnectionOptions`
  - `Endpoint/CpuType/Rack/Slot`
  - `IoPollIntervalMs`
  - 自动重连窗口（最小/最大延迟、退避因子）
- 点位映射：`SiemensS7PointBindingOptions`
  - `PointId`
  - `Binding`（`Area + DbNumber + ByteOffset + BitIndex + TriggerState`）

该方式把业务点位语义与 PLC 地址语义解耦，能显著降低后续 PLC 地址调整的改动面。

## 3.3 初始化 + 监控线程模型

`SiemensS7EmcController` 的运行模型可归纳为：

1. 初始化阶段建立连接，更新控制器状态。
2. 启动周期监控循环（基于 `IoPollIntervalMs`）。
3. 按监控点集合构建批量读取计划（分段读取减少往返）。
4. 将读取结果写回内存快照 `MonitoredIoPoints`。
5. 对外只暴露快照集合，监控消费者不直接访问 PLC。

这是典型“单生产者（监控线程）+ 多消费者（IoPanel/Sensor/HostedService）”模型，吞吐和稳定性都优于多处直接读 PLC。

## 3.4 写 IO 路径

`WriteIoAsync(point, state, writeContext)` 关键步骤：

1. 使用 `PointId` 在绑定表中定位可写地址。
2. 将逻辑电平映射到目标 bit 写入。
3. 带 `writeContext` 透传上下文，便于故障日志定位链路。
4. 写入失败触发 `Faulted` 事件并记录日志。

与读路径解耦，写入按点位执行，读取由监控线程统一维护快照。

## 3.5 重连与故障隔离

控制器内建自动重连退避策略，并通过事件上抛异常状态；上层服务不需要自定义并发重连器，避免多点重连互相干扰。

## 4. ISensorManager（SiemensS7Sensor）实现机制拆解

## 4.1 抽象契约层（Core.Manager.Sensor）

`ISensorManager` 提供：

- `StartMonitoringAsync` / `StopMonitoringAsync`
- `Sensors`（当前传感器状态快照）
- `Status` / `IsMonitoring`
- 事件：`SensorStateChanged`、`MonitoringStatusChanged`、`Faulted`

说明传感器管理被设计为独立子系统，不直接绑死在 IO 面板实现里。

## 4.2 启动前校验与点位同步

`SiemensS7Sensor` 在启动时做两件关键事：

1. 校验 `SensorOptions.PointId` 必须存在于 `SiemensS7PointBindingOptions`。
2. 调用 `SensorWorkflowHelper.SyncMonitoredIoPointsToEmcAsync` 将传感器点位同步给 EMC。

这确保 EMC 监控线程已覆盖传感器需要的点位，避免“传感器在读，EMC 未监控”的空快照问题。

## 4.3 监控循环与去抖

传感器监控循环核心逻辑：

1. 从 `_emcController.MonitoredIoPoints` 取快照。
2. 仅处理配置中存在的点位。
3. 对每个点维护运行态（上次状态、上次变更时间）。
4. 首次采样只建基线，不触发事件。
5. 状态变化后通过去抖窗口判定，再触发 `SensorStateChanged`。

去抖策略由 `SensorWorkflowHelper.PassDebounce` 提供统一实现，减少重复代码与行为漂移。

## 4.4 故障与状态机

`SiemensS7Sensor` 对外维持 `Stopped/Monitoring/Faulted` 状态，任何循环异常都会：

- 记录日志；
- 触发 `Faulted`；
- 切到 `Faulted` 状态。

该策略让 Host 层可基于状态做告警与运维可视化。

## 5. Host 编排模式（为何能稳定运行）

`IoMonitoringHostedService` 的启动顺序具备明确依赖关系：

1. 启动延迟（给硬件与网络预热）。
2. `IEmcController.InitializeAsync`。
3. 下发按钮点位至 EMC。
4. 等待首轮快照。
5. 再启动 `IIoPanel` 与 `ISensorManager`。

这条顺序避免了“消费者先启动，但生产者快照未就绪”的竞态。

DI 层由 `WebApplicationBuilderSiemensS7Extensions` 统一注册：

- Vendor 专属 options + validators
- `IEmcController -> SiemensS7EmcController`
- `ISensorManager -> SiemensS7Sensor`
- `IIoPanel -> SiemensS7IoPanel`

此处体现了清晰的“Core 抽象 + Vendor 实现 + Host 组装”边界。

## 6. 对当前仓库的可迁移结论

可直接复用的设计原则：

1. **统一快照源**：所有 IO 读取集中在 EMC 控制器，业务组件只消费快照。
2. **逻辑点位映射**：通过 `PointId -> S7 binding` 做地址解耦。
3. **先校验后启动**：传感器点位缺失绑定时快速失败并落盘日志。
4. **分层解耦**：Core 仅定义接口与事件；Drivers 承担协议实现；Host 只编排生命周期。
5. **异常隔离**：事件回调异常不反向污染主流程，全部记录日志。

## 7. 三个 PR 的落地实施计划（控制在 3 个拉取请求 内）

## 拉取请求-1：打基础（抽象 + 配置 + 校验）

目标：先把 SiemensS7 接入所需的“协议无关骨架”补齐，不引入真实读写行为。

交付项：

1. Core 层新增/补齐接口与模型（若已有则仅对齐差异）：
   - EMC 控制器抽象（状态、故障、监控点集合、初始化/重连/写入）
   - 传感器管理抽象（监控状态、传感器列表、状态变更/故障事件）
2. Drivers 层新增 SiemensS7 options（连接参数、点位绑定参数）。
3. Host 层新增 options 绑定与 validator 注册入口（仅注册，不启用真实驱动）。
4. 完成配置校验规则：
   - PointId 唯一
   - 地址合法性（Area/Db/Byte/Bit）
   - 传感器点位必须已绑定，且禁止映射到输出区。

验收标准：

- 编译通过；
- 配置错误可在启动前被明确拦截；
- 无业务行为变更。

---

## 拉取请求-2：实现 IEmcController（SiemensS7 驱动核心）

目标：落地 SiemensS7 的 PLC 连接、批量轮询快照、按点写入、自动重连。

交付项：

1. 实现 `SiemensS7EmcController`：
   - 初始化/重连状态机；
   - 周期轮询读取监控点；
   - 内存快照更新（线程安全）；
   - `WriteIoAsync` 点位写入；
   - `Faulted/StatusChanged/Initialized` 事件发布。
2. 引入 Host 启动编排中的 EMC 初始化流程。
3. 补齐单元测试/集成测试（至少覆盖）：
   - 监控点注册与快照更新；
   - 写入路径点位映射；
   - 断线重连后恢复监控。

验收标准：

- EMC 快照持续更新；
- 写入可用；
- 断线可恢复；
- 异常路径全部有日志。

## 拉取请求-3：实现 ISensorManager + IO 监控联动

目标：让传感器监控完全消费 EMC 快照，实现稳定事件流与 Host 端联动。

交付项：

1. 实现 `SiemensS7Sensor`：
   - 启动前点位绑定校验；
   - 向 EMC 同步监控点；
   - 周期采样 + 去抖；
   - 状态变化事件与故障事件。
2. 完成 `IoMonitoringHostedService` 的最终编排闭环：
   - EMC 初始化 -> 点位下发 -> IoPanel/Sensor 启动；
   - 停机顺序与资源释放。
3. 完成联调清单：
   - 传感器状态变化时序正确；
   - 去抖生效；
   - 快照为空与异常路径均可观测。

验收标准：

- 传感器事件可稳定触发且不抖动；
- 服务启停稳定；
- 监控链路具备可观测性与可诊断性。

## 8. 风险与控制

1. **配置风险**：点位映射错误导致误读/误写。  
   控制：拉取请求-1 强校验 + 启动前失败。

2. **并发风险**：多组件并发读 PLC 导致竞争。  
   控制：统一由 EMC 轮询，其他组件仅读快照。

3. **重连风暴风险**：多处重连逻辑叠加。  
   控制：只保留 EMC 内部统一重连策略。

4. **事件异常反噬主流程风险**：回调抛异常影响监控线程。  
   控制：事件回调统一 try/catch 并落盘日志。

---

## 9. 建议的实施顺序与里程碑

1. 先完成拉取请求-1（接口/配置/校验），冻结配置结构。
2. 再推进拉取请求-2（EMC 核心），先打通“可读可写可重连”。
3. 最后拉取请求-3（Sensor + Host 编排），完成业务闭环。

按该顺序可最大化降低返工概率，并将风险集中在可测试阶段逐步消化。
