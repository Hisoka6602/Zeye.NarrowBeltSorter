# 上车匹配时序偏移详细分析与实时计划

## 1. 问题定义

当前上车匹配依赖“当前感应位小车编号 + 上车位固定偏移”进行计算。  
在 `NonFirstCarSensor` 触发后到“包裹实际成熟并尝试上车”之间存在时间差，若该时间差接近一个小车步距周期，则可能出现匹配到上一辆或下一辆小车的错位风险。

## 2. 理论建模与成立条件

### 2.1 基本变量

- 小车步距：`PitchMm`（毫米）
- 环线速度：`SpeedMmps`（毫米/秒）
- 单步距时间：`CarrierPeriodMs = PitchMm / SpeedMmps * 1000`
- 触发后到匹配计算时的有效延迟：`EffectiveDelayMs`
- 延迟占比：`DelayRatio = EffectiveDelayMs / CarrierPeriodMs`

### 2.2 样例验证

- `PitchMm = 300`
- `SpeedMmps = 1000`
- `CarrierPeriodMs = 300ms`

当 `EffectiveDelayMs = 290ms` 时，`DelayRatio≈96.7%`，系统已接近换车边界；再增加少量延迟即可跨车。  
因此“会出现错位风险”的理论成立；但“任何 290ms 场景都必然错位”不严格成立，是否错位取决于实时速度波动、事件处理抖动、采样点位与边沿时序。

## 3. 结论

1. 该理论在工程上成立，且在高速度、小步距、密集包裹场景下风险显著放大。  
2. 仅依赖固定 `LoadingZoneCarrierOffset`，无法覆盖“触发后时间推进导致的位置漂移”。  
3. 需要引入“时间位移补偿”能力，并以实时速度与步距周期计算补偿偏移。  
4. 若不接入实时速度，仅使用固定速度假设，补偿结果在速度波动场景会快速失真，工程收益有限。

## 4. 配置归属建议

不建议将“上车匹配补偿策略”放在 `appsettings.looptrack.json`。  
建议归属到分拣任务时序域（`SortingTask:Timing`）：

- 该策略本质是分拣上车匹配规则，不是轨道控制规则。
- 可避免环轨配置与分拣策略耦合，保持层级边界清晰。

建议新增配置（示意）：

- `CarrierPitchMm`：小车步距（单位：毫米）
- `LoadingMatchForwardShiftThresholdPercent`：延迟占比阈值（单位：百分比，超过后向前补偿 1 车）
- `EnableLoadingMatchTimeCompensation`：是否启用时间补偿
- `LoadingMatchCompensationDelta`：补偿方向与步数（可正可负）
  - 配置正数：在“基准上车小车号”基础上沿前进方向增加步数
  - 配置负数：在“基准上车小车号”基础上沿前进方向反向减少步数
  - 示例（小车总数 100，`LoadingZoneCarrierOffset=8`，感应位小车=`1`）：
    - 基准上车小车号：`1 + 8 = 9`
    - `LoadingMatchCompensationDelta=+2`：目标小车号为 `11`
    - `LoadingMatchCompensationDelta=-2`：目标小车号为 `7`
    - `LoadingMatchCompensationDelta=-10`：按环线回绕后目标小车号为 `99`
- `LoadingMatchCompensationEnterPercent` 与 `LoadingMatchCompensationExitPercent`：补偿启用/退出阈值（用于滞回）
- `LoadingMatchSmoothingWindowSize`：短窗口平滑样本数（用于阈值附近抖动抑制）
- `SensorEventChannelCapacity`：传感器事件通道容量（默认必须加大，避免关键触发 DropWrite）

## 5. 实时计划（可执行）

### 阶段一（观测闭环先观测后改匹配）

- [ ] 记录 `EffectiveDelayMs`、`DelayRatio`、`CarrierPeriodMs` 到上车链路日志
- [ ] 记录上车与落格日志的 `LoopTrackRealtimeSpeedMmps` 字段，形成可追溯闭环（日志值最多两位小数，整数不带小数位）
- [ ] 补充“补偿前后目标小车号”日志，明确 `LoadingMatchCompensationDelta` 生效方向与回绕结果
- [ ] 建立“占比区间分桶”统计（如 0~50%、50~80%、80~95%、95%+）
- [ ] 用现场数据验证“错位事件”与高占比区间的相关性

### 阶段二（最小侵入补偿）

- [ ] 新增分拣时序配置：步距、阈值百分比、补偿开关
- [ ] 在上车匹配点计算 `DelayRatio`
- [ ] `CarrierPeriodMs` 必须按实时速度计算（速度源不准确会直接导致误补偿）
- [ ] 时间戳口径统一：链路时间以 `NonFirstCarSensor` 触发时间透传，禁止中途改用其他时间基准
- [ ] 当 `DelayRatio` 超阈值时，对上车匹配结果执行“向前 +1 车”补偿
- [ ] 仅在开关开启时生效，默认关闭以支持灰度

### 阶段三（灰度与回归）

- [ ] 在测试/预发先启用补偿开关，观察错位率变化
- [ ] 对比启用前后“上车成功率、错位率、告警率”三项指标
- [ ] 阈值附近抖动治理：使用滞回（例如启用 95%，退出 90%）或短窗口平滑，避免 94%~97% 反复抖动导致“时补时不补”
- [ ] 若收益稳定，再逐步扩大启用范围

### 阶段四（持续优化）

- [ ] 引入速度波动保护：低速或速度异常时降级为固定偏移
- [ ] 引入多档补偿策略（后续可扩展，不在首版启用）
- [ ] 形成标准化验收口径并固化到运维巡检项

## 6. 风险与边界

- 禁止引入触发重放机制，并保持“先有包裹再处理触发”的既有原则。  
- 补偿逻辑仅改变上车目标小车选择，不改变包裹创建与触发有效性判定。  
- 首版建议仅支持“0 或 +1 车”补偿，并控制复杂度与风险。
- 实时速度不可用时应明确记录降级日志，禁止无标识回退为静态速度。
- 相关日志中的实时速度值必须统一为“最多两位小数，整数取整”的输出规则，禁止混用不同精度口径。
- 当 `DelayRatio` 在阈值附近波动时，必须启用滞回或短窗口平滑，避免补偿状态频繁翻转。
- 传感器事件通道满载导致 `DropWrite` 时，补偿算法无法修复“关键触发已丢”引发的相位跳变，默认容量必须加大并可配置。
- 时间戳口径不一致会破坏补偿判定，必须以 `NonFirstCarSensor` 触发时间贯穿创建、上车、落格链路。
