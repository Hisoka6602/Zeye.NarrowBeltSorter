# 包裹密集导致上车小车号偏差分析

## 1. 问题定义

现场现象：当包裹创建更密集时，“上车位小车号”看起来更容易偏差。  
本文仅基于当前仓库代码做链路分析，不引入代码修改方案。

---

## 2. 结论（先给结论）

“包裹越密集，上车小车号越容易错误”的核心原因不是单一公式错误，而是**高频触发下的时序抖动被放大**：

1. 感应位编号更新由传感器事件触发后异步投递；
2. 上车动作消费的是“触发时刻可见到的当前感应位编号”；
3. 包裹密集时，事件队列与线程调度抖动增大，编号更新与包裹上车的相对先后更容易错位；
4. 一旦错位，`LoadingZoneCarrierOffset` 的环形偏移会把这次偏差直接投射为“上车位小车号偏差”。

---

## 3. 代码链路拆解（按时间顺序）

### 3.1 感应位编号来源

- `CarrierLoopGroupingHostedService` 在首车/非首车触发时计算 `currentCarrierId`，并调用：
  - `_carrierManager.UpdateCurrentInductionCarrierAsync(currentCarrierId)`  
  代码位置：`Zeye.NarrowBeltSorter.Execution/Services/CarrierLoopGroupingHostedService.cs`

该调用通过 `_safeExecutor.ExecuteAsync(...)` 异步投递，不是严格串行等待完成后再处理下一次触发。

### 3.2 当前感应位编号事件

- `InfraredSensorCarrierManager.UpdateCurrentInductionCarrierAsync` 更新 `CurrentInductionCarrierId` 并发布 `CurrentInductionCarrierChanged`。  
  代码位置：`Zeye.NarrowBeltSorter.Execution/Services/Carrier/InfraredSensorCarrierManager.cs`

事件时间 `ChangedAt` 取 `DateTime.Now`（本地时间），表示“管理器更新时间点”，不等于原始传感器采样时刻。

### 3.3 上车位计算

- `SortingTaskDropOrchestrationService.HandleCurrentInductionCarrierChangedAsync` 收到当前感应位变化后，调用：
  - `_carrierLoadingService.TryLoadParcelAtLoadingZoneAsync(currentInductionCarrierId, changedAt, ...)`
  代码位置：`Zeye.NarrowBeltSorter.Execution/Services/SortingTaskDropOrchestrationService.cs`

- `SortingTaskCarrierLoadingService.TryLoadParcelAtLoadingZoneAsync` 用以下逻辑计算上车位小车号：
  - `loadingCarrierId = CircularValueHelper.GetCounterClockwiseValue(currentInductionCarrierId, LoadingZoneCarrierOffset, totalCarrierCount)`
  代码位置：`Zeye.NarrowBeltSorter.Execution/Services/SortingTaskCarrierLoadingService.cs`

这意味着：只要 `currentInductionCarrierId` 相对真实瞬时位置落后/超前 1~N 个编号，最终上车位小车号也会按同样偏差被映射。

---

## 4. 为什么“包裹密集”会放大偏差

### 4.1 高频触发导致异步链路更易积压

包裹更密集通常对应更高频的传感器触发。  
在异步投递模型下，触发间隔缩短会提高以下概率：

- 上一次编号更新尚未完成，下一次触发已到达；
- 编号更新事件与上车尝试事件在调度上交错；
- 某一时刻上车计算读取到“前一个触发的编号”。

### 4.2 队列有货时，错位会立刻变成“错误上车号”

`TryLoadParcelAtLoadingZoneAsync` 会先检查待上车队列；队列非空时即按当前可见编号执行映射和装车。  
包裹越密集，待上车队列越容易持续非空，时序错位一旦发生就更容易直接体现在业务结果中。

### 4.3 事件时间语义差异会增加排障体感

当前链路中记录的 `ChangedAt` 是管理器更新时间，不是传感器原始采样时间。  
在高频场景下，即便绝对延迟只有毫秒级，也可能在日志上表现为“编号更新晚于预期”，增加“看起来越来越不准”的体感。

## 5. 仅分析阶段可执行的验证建议（不改代码）

1. 统计“传感器触发时间”与“CurrentInductionCarrierChanged 发布时间”的时间差分布（P50/P95/P99）。
2. 统计“包裹到达上车位事件”发生时，最近一次 `CurrentInductionCarrierId` 变更距离（毫秒差与编号差）。
3. 分密度区间（低/中/高）对比上车偏差率，验证是否随触发频率上升而单调变差。
4. 校验 `LoadingZoneCarrierOffset` 与现场机械相位是否一致，避免把固定偏移误差误判为密度导致问题。
