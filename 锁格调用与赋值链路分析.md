# 锁格调用与赋值链路分析

## 1. 锁格能力定义位置

- 接口定义：`Zeye.NarrowBeltSorter.Core/Manager/Chutes/IChuteManager.cs`
  - `SetChuteLockedAsync(long chuteId, bool isLocked, ...)`
  - `LockedChuteIds`
  - `ChuteLockStatusChanged`

## 2. 当前调用链路（生产代码）

### 2.1 直接调用 `SetChuteLockedAsync` 的生产入口

- 当前生产代码中未发现直接调用 `SetChuteLockedAsync` 的路径。
- 代码检索结果显示，生产层仅保留方法实现，调用主要来自测试代码与驱动设计文档。

### 2.2 与“关闭格口”相关的实际链路

- `Zeye.NarrowBeltSorter.Execution/Services/ChuteSelfHandlingHostedService.cs`
  - `TryCloseAllChutesAsync(...)` 已改为调用 `SetForcedChuteSetAsync(Array.Empty<long>(), ...)`
  - 该链路作用是“全部断开强排输出”，不会写入锁格集合。

## 3. 锁格状态赋值链路（核心实现）

实现位置：`Zeye.NarrowBeltSorter.Drivers/Vendors/ZhiQian/ZhiQianChuteManager.cs`

### 3.1 写入入口

- `SetChuteLockedAsync(chuteId, isLocked, ...)`
  1. 校验连接状态与格口映射
  2. 进入 `_writeLock` 串行化写操作
  3. 读取旧值：`_lockedChuteIds.Contains(chuteId)`
  4. 当 `isLocked=true`：
     - 下发 DO 断开：`_adapter.WriteSingleDoAsync(doIndex, false, ...)`
     - 赋值：`_lockedChuteIds.Add(chuteId)`
  5. 当 `isLocked=false`：
     - 赋值：`_lockedChuteIds.Remove(chuteId)`
  6. 发布事件：`ChuteLockStatusChanged`

### 3.2 读取与冲突校验链路

- `IsChuteLocked(chuteId)` 读取 `_lockedChuteIds`
- 强排路径会读取该状态并拒绝冲突操作：
  - `SetForcedChuteAsync(...)`
  - `SetForcedChuteSetAsync(...)`

## 4. 默认值链路（默认不锁格）

- `ZhiQianChuteManager` 初始化时：`_lockedChuteIds = new HashSet<long>()`
- 配置文件中无“启动即锁格”字段。
- 关闭格口流程已改为“清空强排集合”，不再调用“锁格=true”。
- 因此默认语义为：格口初始状态不锁格。

## 5. 测试层调用链路

- `Zeye.NarrowBeltSorter.Core.Tests/ZhiQianChuteManagerTests.cs`
  - 覆盖 `SetChuteLockedAsync(true/false)`
  - 覆盖锁格与强排冲突保护（锁格后强排失败）
  - 覆盖 `LockedChuteIds` 与 `ChuteLockStatusChanged` 断言
