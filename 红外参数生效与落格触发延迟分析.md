## 2. 结论先说

### 2.1 你配置的参数是否“真正生效”

结论：**参数会进入下发链路，但“是否真实下发成功”当前日志有误判风险；并且部分字段本身并不会参与编码。**

主要原因：

1. 参数下发链路确实存在：
   - 热更新服务会读取 `InfraredChuteOptionsMap` 并逐格口调用 `WriteInfraredChuteOptionsAsync`。
2. 编码器有严格范围校验，超限会直接失败（例如速度过大）。
3. 但 `ZhiQianChute.WriteInfraredChuteOptionsAsync` 当前返回的是 `execution.Success`，不是业务结果 `execution.Result`。这会导致“编码失败返回 false 但无异常”时，外层仍可能判定成功并打“热更新成功”日志。
4. `HoldDurationMs` 当前未参与任何协议帧编码，修改它通常不会带来实际设备动作变化。

### 2.2 为什么“感觉变化不大”

综合代码分析，至少有 4 个高概率原因：

1. **配置值超限，实际并未编码下发**（例如示例中的 `DefaultSpeedMmps=5000` 在当前参数组合下已超 7bit 编码上限）。
2. **日志成功判定存在误导**（见上面的 `execution.Success` 与 `execution.Result` 差异）。
3. **部分参数当前未使用**（`HoldDurationMs`）。
4. **系统落格触发主要由“到位判定 + 格口开闭链路”决定，不完全由红外参数决定。**

### 2.3 为什么日志里 CarrierId 看着有些“错”，但仍能正确落格

结论：**当前 CarrierId 更偏“系统内逻辑编号/映射编号”，与现场人工认知的物理小车号可能不一致；但只要映射链路自洽，仍可正确落格。**

关键点：

1. `CarrierLoopGroupingHostedService` 建环期间用触发计数生成车号（`GetCurrentCarrierId` 直接返回 triggerIndex），本质是逻辑编号序列。
2. 后续装车、到位、落格都基于同一套逻辑编号 + `ChuteCarrierOffsetMap` 计算，因此内部一致时会正确落格。
3. 你看到“日志车号怪异”的现象，更多是“展示编号语义”和“现场物理编号语义”不一致。

### 2.4 为什么感觉开闭格口指令发送有延迟（小车离开很远才触发）

结论：**这是“检测触发点 + 事件调度 + 串行通信门控 + 轮询竞争”叠加的系统性延迟，不是单点问题。**

最重要的延迟来源：

1. 触发基准来自感应位事件，不是格口处独立传感器实时直触发。
2. 下发 DO 与读 DO 共用 `_requestGate` 串行门控，轮询读取会与写入竞争。
3. 存在 `CommandAbsoluteIntervalMs` 发送绝对间隔（默认 20ms）。
4. `DropAsync` 内部还有固定开关间隔 `350ms`（这是开后等待关，不是触发前等待，但会影响体感节奏）。
5. 若 `ChuteCarrierOffsetMap` 偏移未按现场校准，会表现为“总是早/晚几车甚至更远”。

## 3. 红外参数生效链路（按代码时序）

1. `ChuteSelfHandlingHostedService` 启动后/配置变更后调用 `TryApplyInfraredOptionsAsync`。
2. 服务遍历 `InfraredChuteOptionsMap`，逐个格口调用 `IChute.WriteInfraredChuteOptionsAsync(...)`。
3. `ZhiQianChute.WriteInfraredChuteOptionsAsync`：
   - 调用 `LeadshaineInfraredDriverFrameCodec.EncodeAsync` 进行 8 字节帧编码。
   - 编码成功后通过 `ZhiQianBinaryClientAdapter.WriteInfraredFrameAsync` 透传发送。
4. 发送链路与 DO 读写共用 `_requestGate` 串行门控。

## 4. 各参数上下限与当前是否生效

> 下表是“当前代码实际限制”，优先级高于配置注释文字。

| 参数 | 当前代码是否参与设备帧 | 代码约束/上下限（当前实现） | 备注 |
|---|---|---|---|
| `DinChannel` | 是 | 仅支持 `1` 或 `2`（映射 D1/D2） | 其他值编码失败 |
| `DefaultDirection` | 是 | 枚举值（Left/Right） | 仅影响 Byte2 方向位 |
| `ControlMode` | 是 | `Time` / `Position` | 影响 Byte6 模式位与 runRaw 解释 |
| `DefaultSpeedMmps` | 是 | `>0` 且换算后 `speedRaw` 必须在 `0..127` | 超限编码失败；不是“自动截断” |
| `DefaultDistanceMm` | 是（Position 模式） | `>0` 且换算后 `runRaw` 在 `0..255` | Position 模式必填 |
| `DefaultDurationMs` | 是（Time 模式） | `>0` 且换算后 `runRaw` 在 `0..255` | Time 模式必填 |
| `AccelerationMmps2` | 是 | **1500 ~ 7850**（步进 50） | 超限编码失败 |
| `TriggerDelayMs` | 是 | `>=0` 且换算后 `delayRaw` 在 `0..255` | 超限编码失败 |
| `RollerDiameterMm` | 是 | `>0` | 参与速度/距离换算 |
| `DialCode` | 是 | 入参是 byte，但实际只取低 6 bit（`0..63`） | `>63`会被掩码截断 |
| `HoldDurationMs` | **否** | 当前未进入编码帧 | 改动通常无设备效果 |

### 4.1 结合示例（102）快速计算

示例：`RollerDiameterMm=67`、`DefaultSpeedMmps=5000`

- 当前速度换算：
  - `rpm = DefaultSpeedMmps * 60 / (π * D)`
  - `speedRaw = rpm / 6`
- 当 `D=67` 时，`speedRaw` 超过 127（上限），会编码失败。
- 对应可编码的近似最大线速度约为 **2672 mm/s**（按当前代码常量计算）。推导为：`Vmax = speedRawMax * RpmPerSpeedRaw * ProtocolPi * D / 60 = 127 * 6 * 3.14 * 67 / 60 ≈ 2672 mm/s`。其中 `3.14` 是协议编码常量 `ProtocolPi` 的固定取值（协议值，不是数学常数 `Math.PI`）。

因此示例中的 `5000` 在当前实现中大概率不会被成功编码下发。

## 5. 为什么“参数改了但体感没变化”

### 5.1 编码失败但日志可能看起来成功

`WriteInfraredChuteOptionsAsync` 内部返回值判断是 `execution.Success`：

- `execution.Success=true` 仅代表“委托执行过程无异常”。
- 编码器返回 `false`（例如参数超限）时，不抛异常也可能仍被外层当成功。

这会造成“日志看着已更新，但现场效果没变化”的错觉。

### 5.2 部分参数本身不生效

`HoldDurationMs` 在当前仓库只用于配置签名拼接，不参与下发帧。

### 5.3 实际动作主导因素并非只有红外参数

落格触发主要受：

- 感应位事件到达时机
- `ChuteCarrierOffsetMap` 偏移标定
- 通信串行门控竞争
- 线程调度

影响，红外参数只是链路中的一环。

---

## 6. CarrierId 看起来“错”但仍落格正确的原因

1. 建环期间，车号来自触发序列计数，属于逻辑编号语义。
2. 装车绑定和落格命中都沿用同一逻辑编号体系。
3. 只要这套编号与偏移映射自洽，系统就能“逻辑正确落格”。
4. 现场若用的是另一套物理标识（喷码/贴标号），就会出现“日志看起来错号”的观感。

---

## 7. 开闭格口触发“感觉慢”的代码级原因拆解

按链路从前到后拆解：

1. **事件触发基准**：来自感应位，不是格口位。
2. **事件调度方式**：通过 `SafeExecutor.PublishEventAsync` 和线程池异步派发，存在调度抖动。
3. **落格判定计算**：在 `SortingTaskDropOrchestrationService` 里遍历偏移表并命中目标格口。
4. **通信串行门控**：
   - `ReadDoStatesAsync`（轮询）与 `WriteSingleDoAsync`（开闭闸）共用 `_requestGate`。
   - 正在读状态时，写命令需排队等待。
5. **发送间隔限制**：`CommandAbsoluteIntervalMs` 约束发送节奏。
6. **执行动作时序**：`DropAsync` 里先开闸，再等待 `350ms` 后关闸。

### 7.1 最优先排查项（不改代码前提）

1. 校准 `Carrier:ChuteCarrierOffsetMap` 是否与现场几何位置一致。
2. 观察轮询读命令与开闸写命令时间线是否发生排队（同一时段 `_requestGate` 竞争）。
3. 检查 `appsettings*.json` 中：
   - `Chutes:ZhiQian:PollIntervalMs`
   - `Chutes:ZhiQian:CommandAbsoluteIntervalMs`
4. 检查感应器安装点是否天然晚于期望触发点（感应位置偏后会导致“看起来晚开闸”）。

## 8. 建议的现场验证清单（仅运维/参数侧）

1. 先把待验证格口（如 102）的参数改到“肯定可编码范围内”。

## 9. 本文对你两个问题的直接回答

### 问题 1：参数是否真正生效、上下限是多少

- 会进入下发链路，但当前存在“失败被误判成功”的风险。
- 上下限已在第 4 节给出（按当前代码真实限制）。
- 示例中的 `102.DefaultSpeedMmps=5000` 按现有实现很可能超限，不会成功编码下发。

### 问题 2：为什么 CarrierId 有时看着错但仍正确落格

- 因为日志中的 CarrierId 使用的是系统逻辑编号体系，不一定等于现场物理编号。
- 落格依赖的是同一套内部映射，内部一致就能正确落格。

### 新增问题：为什么开闭格口看起来有延迟

- 主要是“感应触发点 + 偏移映射 + 串行门控 + 轮询竞争 + 发送间隔”叠加，而非单点故障。
