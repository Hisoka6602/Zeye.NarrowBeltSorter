# 雷赛红外参数边界与实时性链路排查

## 1. 目的与范围

本文用于沉淀以下内容：

1. 红外参数在当前代码中的参与性、边界与换算公式（含 `RollerDiameterMm=67` 基准值）。
2. `上车位装车成功 CarrierId=...` 的链路语义校验。
3. “小车离开很远才触发”在当前链路中的非 `ChuteCarrierOffsetMap` 因素排查。
4. 当前还能配置且会生效的参数清单。

本文结论仅基于仓库当前实现，不包含未落地分支代码。

---

## 2. 红外参数边界（`RollerDiameterMm=67` 基准）

> 说明：以下上下限是“当前代码编码可通过的范围”，不是设备手册理论极限。

| 参数 | 最小值 | 最大值 | 说明 |
|---|---:|---:|---|
| `DinChannel` | 1 | 2 | 当前编码仅支持 D1/D2 |
| `DefaultDirection` | Left | Right | 枚举 |
| `ControlMode` | Time | Position | 枚举 |
| `DefaultSpeedMmps` | `>0` | `≈2672 mm/s` | `speedRaw` 必须在 `0..127` |
| `DefaultDistanceMm`（Position） | `>0` | `≈2575 mm` | `runRaw` 必须在 `0..255` |
| `DefaultDurationMs`（Time） | `>0` | `2550 ms` | `runRaw = durationMs / 10` |
| `AccelerationMmps2` | 1500 | 7850 | 且按 50 步进换算 |
| `TriggerDelayMs` | 0 | 2550 | `delayRaw = delayMs / 10`，`0..255` |
| `RollerDiameterMm` | `>0` | 代码未设上限 | 参与速度/距离换算 |
| `DialCode` | 0 | 63（有效位） | 超出后按低 6 bit 掩码 |
| `HoldDurationMs` | 任意 int | 任意 int | 当前不参与帧编码 |

### 2.1 换算公式（当前实现）

基于 `LeadshaineInfraredDriverFrameCodec` 常量：

- `ProtocolPi = 3.14`
- `RpmPerSpeedRaw = 6`
- `CirclePerPositionRaw = 0.048`
- `DelayTickMs = 10`
- `DurationTickMs = 10`

#### 速度（`DefaultSpeedMmps -> speedRaw`）

1. `circumference = RollerDiameterMm * ProtocolPi`
2. `rpm = DefaultSpeedMmps * 60 / circumference`
3. `speedRaw = round(rpm / 6)`（四舍五入，超 `0..127` 失败）

当 `RollerDiameterMm=67` 时：

- `Vmax = 127 * 6 * 3.14 * 67 / 60 ≈ 2672 mm/s`

#### 距离（Position 模式，`DefaultDistanceMm -> runRaw`）

1. `circumference = RollerDiameterMm * ProtocolPi`
2. `circles = DefaultDistanceMm / circumference`
3. `runRaw = round(circles / 0.048)`（超 `0..255` 失败）

当 `RollerDiameterMm=67` 时：

- `Dmax = 255 * 0.048 * 3.14 * 67 ≈ 2575 mm`

#### 时间（Time 模式，`DefaultDurationMs -> runRaw`）

- `runRaw = round(DefaultDurationMs / 10)`（超 `0..255` 失败）
- 对应 `DefaultDurationMs` 最大值 `2550 ms`

#### 延时（`TriggerDelayMs -> delayRaw`）

- `delayRaw = round(TriggerDelayMs / 10)`（超 `0..255` 失败）
- 对应 `TriggerDelayMs` 最大值 `2550 ms`

#### 加速度（`AccelerationMmps2 -> accelerationRaw`）

- `accelerationRaw = round((AccelerationMmps2 - 1500) / 50)`，且 `AccelerationMmps2` 必须在 `1500..7850`

## 3. “上车位装车成功 CarrierId=...” 语义核对

`SortingTaskCarrierLoadingService.TryLoadParcelAtLoadingZoneAsync` 的 `CarrierId` 计算为：

1. 输入 `currentInductionCarrierId`（感应位当前小车号）；
2. 使用 `LoadingZoneCarrierOffset` 通过 `CircularValueHelper.GetCounterClockwiseValue` 计算上车位编号；
3. 日志 `上车位装车成功 CarrierId=...` 输出的是第 2 步得到的上车位小车号。

因此：

- `ChuteCarrierOffsetMap` 只参与“格口命中”判定，不参与“上车位装车”编号计算。
- 若该日志出现十几个小车量级偏差，优先排查“感应位当前小车号”本身来源链路，而不是先改 `ChuteCarrierOffsetMap`。

## 4. “离开很远才触发”链路中除 `ChuteCarrierOffsetMap` 外的风险点

### 4.1 感应位小车号更新是异步投递

`CarrierLoopGroupingHostedService` 在传感器触发后，异步调用 `_carrierManager.UpdateCurrentInductionCarrierAsync(...)`。  
该更新不是与传感器采样同线程串行直传，存在线程调度抖动窗口。

### 4.2 事件时间戳使用 `DateTime.Now`，不是传感器 `OccurredAtMs`

`InfraredSensorCarrierManager.UpdateCurrentInductionCarrierAsync` 生成 `CurrentInductionCarrierChangedEventArgs.ChangedAt = DateTime.Now`。  
如果调度晚于采样点，链路日志会体现为“触发偏晚”。

### 4.3 触发源采样周期与去抖窗口会放大延迟

`LeadshaineSensorManager` 对每个传感器支持：

- `PollIntervalMs`（单点轮询周期）
- `DebounceWindowMs`（去抖窗口）

二者越大，触发越晚。

### 4.4 落格执行链路与开关动作本身存在固定时序

`SortingTaskDropOrchestrationService` 在命中后执行 `DropAsync(parcel, droppedAt, 350ms)`。  
该 350ms 是开闸后到关闸的动作时窗，不是触发判定前等待，但会影响现场体感节奏。

### 4.5 格口通信链路存在门控与节流

`ZhiQianBinaryClientAdapter` 的读写共用 `_requestGate`，并受 `CommandAbsoluteIntervalMs` 限制。  
在高频轮询与高频写入并发时，写命令会排队。

## 5. 当前可配置参数清单（除 `ChuteCarrierOffsetMap` 外）

### 5.1 红外帧相关（`InfraredChuteOptions`）

- `DinChannel`
- `DefaultDirection`
- `ControlMode`
- `DefaultSpeedMmps`
- `DefaultDurationMs`
- `DefaultDistanceMm`
- `AccelerationMmps2`
- `TriggerDelayMs`
- `RollerDiameterMm`
- `DialCode`
- `HoldDurationMs`（当前仅配置存储，不进帧）

### 5.2 上车与落格映射（`CarrierManagerOptions`）

- `LoadingZoneCarrierOffset`（上车位偏移，直接影响“上车位装车成功 CarrierId”）
- `ChuteCarrierOffsetMap`（格口偏移，仅用于落格命中）

### 5.3 传感器采样与触发（Leadshaine）

- `Leadshaine:EmcConnection:PollingIntervalMs`（默认轮询基准）
- `Leadshaine:Sensors:Sensors[*]:PollIntervalMs`（单点覆盖）
- `Leadshaine:Sensors:Sensors[*]:DebounceWindowMs`

### 5.4 格口通信与动作节奏（ZhiQian）

- `Chutes:ZhiQian:CommandTimeoutMs`
- `Chutes:ZhiQian:RetryCount`
- `Chutes:ZhiQian:RetryDelayMs`
- `Chutes:ZhiQian:CommandAbsoluteIntervalMs`
- `Chutes:ZhiQian:PollIntervalMs`
- `Chutes:ZhiQian:DefaultOpenDurationMs`

## 6. 现场优先排查顺序（针对“离开很远才触发”）

1. 先核对 `LoadingZoneCarrierOffset` 与上车位物理位置，确认“上车位装车 CarrierId”是否跟随感应位一致变化。
2. 采集同一包裹的三段时间：传感器触发、`CurrentInductionCarrierChanged`、`小车到达目标格口准备落格`。
3. 将相关传感器 `PollIntervalMs` 与 `DebounceWindowMs` 调小到稳定下限后复测。
4. 检查 `CommandAbsoluteIntervalMs` 与轮询压力，确认写命令是否被 `_requestGate` 排队。
5. 在上述完成后再微调 `ChuteCarrierOffsetMap`，仅用于修正格口命中几车偏差。
