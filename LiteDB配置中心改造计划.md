# LiteDB 配置中心改造计划

## 1. 改造目标

- 目标一：将业务配置从 `appsettings*.json` 迁移到 LiteDB 持久化存储。
- 目标二：所有业务配置统一通过 API 端点进行新增、修改、删除、查询与生效。
- 目标三：配置提交时必须经过模型校验与业务规则校验，校验失败不落库、不生效。
- 目标四：配置变更后支持热更新，运行中服务无需重启即可感知新配置。
- 目标五：`appsettings.json` 仅保留 `LogCleanup` 基础配置，其他业务配置全部清理。
- 目标六：后续新增/删除/修改配置项必须遵循“API 管理 + LiteDB 持久化 + 热更新”统一流程。

## 2. 范围与边界

### 2.1 本次纳入范围

- Core/Drivers/Execution/Host 当前通过 `IOptions`、`IOptionsMonitor`、配置绑定使用的业务配置。
- 设备参数、分拣参数、联动参数、时序参数、状态策略参数等运行期配置。
- 配置管理 API（查询、变更、校验、回滚、发布）与配置热更新通知链路。

### 2.2 保留范围

- `LogCleanup` 保留在 `appsettings.json`（含中文注释与范围说明）。
- NLog 目标与路由保留现有文件配置方式，不迁移到 LiteDB。

### 2.3 非目标

- 不改变现有业务流程语义（仅替换配置来源与生效机制）。
- 不引入第二套并行配置写入链路，避免双写不一致。

## 3. 目标架构

### 3.1 分层设计

- **配置域模型层（Core.Options）**：保留现有 Options 类型，作为配置契约与校验载体。
- **配置存储层（Infrastructure）**：新增 LiteDB 配置仓储，实现按模块分组、版本化、原子写入。
- **配置应用层（Execution/Host）**：封装配置发布器、热更新分发器、回滚协调器。
- **配置接入层（Ingress）**：提供配置 API 端点（读/改/校验/发布/回滚）。

### 3.2 单一写入入口

- 仅允许 API 端点写入 LiteDB。
- 禁止业务服务直接写 LiteDB 配置集合。
- 配置变更统一经过：请求模型校验 -> 业务规则校验 -> 事务写入 -> 发布热更新事件。

### 3.3 热更新机制

- 配置发布后，按“模块维度”广播变更通知。
- 各消费服务通过非阻塞方式订阅配置变更并并行拉取最新快照。
- 变更失败时记录异常日志并触发模块级回滚策略。

## 4. 数据模型与持久化策略

## 4.1 LiteDB 集合规划

- `config_documents`：配置主体（模块、键、值、版本、状态、更新时间）。
- `config_revisions`：配置历史（快照、变更人、变更说明、发布时间）。
- `config_publish_records`：发布记录（发布批次、结果、失败详情、回滚标记）。

### 4.2 版本与并发控制

- 每个模块配置维护递增版本号。
- API 更新必须带上期望版本，版本不一致则拒绝提交。
- 发布成功后写入发布记录，支持按版本快速回滚。

### 4.3 序列化规范

- 配置值统一使用结构化 JSON 文本存储。
- 时间字段统一本地时间语义，不允许 UTC 语义与 UTC 转换链路。

## 5. API 端点规划

### 5.1 查询端点

- 查询模块当前生效配置。
- 查询模块配置历史版本。
- 查询最近发布记录与失败详情。

### 5.2 变更端点

- 新增模块配置。
- 修改模块配置。
- 删除模块配置（仅允许删除未被依赖或已停用配置）。

### 5.3 校验与发布端点

- 预校验端点：仅执行模型+规则校验，不落库。
- 保存草稿端点：落库但不发布。
- 发布端点：对指定版本执行发布并触发热更新。
- 回滚端点：将模块配置回滚到指定历史版本并重新发布。

## 6. 校验体系

### 6.1 模型校验

- 复用现有 `Options` 校验规则（DataAnnotations/自定义 Validator）。
- 校验错误按字段返回中文错误信息与可填写范围。

### 6.2 业务规则校验

- 跨配置项依赖关系校验（同模块、跨模块）。
- 设备协议边界校验（地址、位号、数量、频率、阈值范围）。
- 系统状态相关约束校验（避免运行中写入非法状态配置）。

### 6.3 发布前校验

- 发布前再次执行全量校验，防止草稿与当前运行环境不一致。
- 任一校验失败，发布终止并记录异常日志。

## 7. 热更新落地策略

### 7.1 配置消费模式

- 各服务改为通过统一配置访问器读取“内存快照”。
- 快照由发布器原子替换，读取端无锁或最小锁开销访问。

### 7.2 变更传播机制

- 按模块发送变更事件，订阅者并行处理，禁止串行阻塞。
- 单个订阅者失败不得影响其他订阅者获取新配置。

### 7.3 故障处理

- 发布后消费失败时，记录错误日志并标记模块降级状态。
- 支持一键回滚到最近稳定版本并再次发布。

## 8. 配置迁移计划

### 8.1 迁移准备

- 盘点全部 `appsettings*.json` 业务配置项及其归属模块。
- 建立“配置项 -> Options 类型 -> 消费服务”映射清单。

### 8.2 迁移执行

- 将业务配置导入 LiteDB 初始版本。
- API 提供迁移校验接口，确认导入后配置可通过全量校验。
- 服务配置读取切换到 LiteDB 快照源。

### 8.3 收口处理

- 从 `appsettings.json` 与模块配置文件移除业务配置，仅保留 `LogCleanup`。
- 更新配置文档与运维手册，统一使用 API 进行配置管理。

## 9. 后续配置变更固定流程（强制）

- 新增配置项：先定义 Options 契约与校验规则，再开放 API 字段，最后接入发布与热更新。
- 修改配置项：必须保持向后兼容策略（默认值/迁移脚本/版本转换）并提供回滚路径。
- 删除配置项：先标记停用并完成消费端清理，再执行版本化删除与历史保留。
- 任意配置变更不得直接写 `appsettings*.json`，必须经 API + LiteDB 流程。
